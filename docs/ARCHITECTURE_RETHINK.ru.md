# VPN Gateway — переосмысление архитектуры (без потери функционала)

Цель: снизить когнитивную сложность, сохранив весь функционал.

## Проблема текущего состояния

Сейчас сложность размазана по:
- нескольким compose-файлам,
- Makefile-алиасам,
- неочевидному разделению «кто владеет peer-конфигами, а кто их только публикует».

Из-за этого сложно быстро ответить на вопросы:
- какой файл запускать,
- где «источник правды»,
- что именно делает net-manager.

---

## Целевая модель (предлагаемая)

### 1) Явные роли подсистем

1. **WireGuard Plane**
   - хранение и жизненный цикл peer-ключей/базовых конфигов (`data/wg`).
2. **Gateway Plane**
   - прозрачный TCP/UDP gateway, proxy pool, API.
3. **Distribution Plane (опционально)**
   - net-manager: UPnP + публикация LAN/WAN конфигов/QR (`data/clients`).

Ключевая идея: Distribution Plane никогда не модифицирует WireGuard Plane.

### 2) Один базовый compose + профили

Вместо нескольких «почти одинаковых» файлов:
- `docker-compose.yml` (base: wireguard + gateway + unbound)
- профиль `nat` включает net-manager + macvlan
- профиль `dev` переключает net-manager в host/bridge

Плюсы:
- меньше дублирования,
- меньше рассинхронизации,
- проще UX: `docker compose --profile nat up -d`.

### 3) Makefile как UX-обертка, а не второй source of truth

Makefile должен лишь вызывать 2–3 канонические команды, без дублирования логики.

---

## План миграции (эволюционный)

### Шаг 1 (документационный, без риска)
- Зафиксировать “Mode Matrix” в отдельном документе.
- Объявить owner каждого каталога данных:
  - `data/wg` owner = wireguard,
  - `data/clients` owner = net-manager.

### Шаг 2 (компоновка)
- Перейти на profile-based compose.
- Сохранить старые make-таргеты как алиасы для обратной совместимости.

### Шаг 3 (операционная наблюдаемость)
- Единый `make status-all`:
  - health gateway,
  - proxy metrics,
  - network-status (если профиль nat).

### Шаг 4 (конфигурационная дисциплина)
- Валидатор `.env` (скрипт preflight):
  - проверка `WG_SERVER_URL` для VPS,
  - проверка `NET_INTERFACE`/`MACVLAN_IP_RANGE` для NAT,
  - проверка конфликтов портов.

---

## Подводные камни, которые надо держать в голове

1. `WG_SERVER_URL=auto` может быть неверным для удаленных клиентов.
2. UPnP не гарантирован на всех роутерах.
3. macvlan ограничен на Docker Desktop.
4. Дублирование compose-логики почти всегда превращается в drift.
5. Отсутствие preflight-проверок ведёт к «тихим» ошибкам на старте.

---

## Решение “здесь и сейчас” (без большого рефакторинга)

Если не хотим трогать runtime прямо сейчас:
1. Оставить текущие compose-файлы.
2. Считать каноническим этот мануал + operation manual.
3. Добавить preflight-скрипт и режимный status-check.

Это снизит сложность использования уже сегодня, даже до архитектурной миграции.
