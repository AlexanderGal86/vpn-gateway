- Parses HTTP status code: only accepts 200 (connection established)
- Returns error for status codes like 403, 407, etc.

**SOCKS5 Protocol** (lines 98-195):

- Response validation: checks REP byte for success (0x00) or error codes (0x01-0x08)
- Reads and discards bind address/port (ATYP=1/3/4 for IPv4/domain/IPv6)

```rust
    let latency_ms = if latency_ms.is_finite() {

    } else {
        5000.0
    };
```

- **Clamping:** Protects against NaN/Infinity; max 60s (60,000ms)

```rust
    self.fail_count += 1;
    self.consecutive_fails += 1;
    
    let disable_secs = match self.consecutive_fails {

        5..=9 => 60,         // 1 minute
        10..=19 => 300,      // 5 minutes
        20..=49 => 3600,     // 60 minutes
```

```rust
score() = latency_ewma + 
          (fail_penalty) + 
          (circuit_breaker_penalty) - 
          priority_boost
```
### **Latency Tracking:**
- **EWMA (α = 0.2)**: `new = old * 0.8 + measured * 0.2`
- Clamped to [0, 60000] ms
- Initial pessimistic: 5000 ms
### **Circuit Breaker:**
- Escalating disable duration after consecutive failures:
  - 0-4 failures: enabled
  - 5-9: disabled 60s
  - 10-19: disabled 300s
  - 20-49: disabled 3600s (1 hour)
  - 50+: permanently failed until next source refresh
### **Verification Stages** (3-stage health check):
**Stage 1 - TCP Connect** (timeout: configurable, default 3000ms for fast probe)
- Tests basic network reachability
- Returns latency
**Stage 2 - HTTP CONNECT Handshake** (to cloudflare.com:80)
- Verifies proxy actually forwards traffic
- Checks for `HTTP/1.1 200` response
- Returns latency
**Stage 3 - TLS Validation** (to cloudflare.com:443)
- Establishes tunnel through proxy (CONNECT or SOCKS5)
- Performs TLS handshake with certificate validation
- Uses system root CA bundle (webpki_roots)
- Detects MITM proxies (self-signed certs)
Result: `tls_clean = true` (clean) or `false` (MITM proxy but works)

## **9. CONCURRENT CONNECTION HANDLING**
### **Max Connections:**
- **Config limit**: `max_connections` (default 10000)
- **Enforcement**: Transparent proxy spawns with semaphore
- **Per-proxy**: Connection pool allows `connection_pool_max_per_proxy` (default 10 idle)
### **Connection Lifecycle:**
1. Client connects → transparent proxy intercepts
2. Gets original destination via `SO_ORIGINAL_DST` (iptables REDIRECT)
3. Checks for sticky session (client IP → proxy mapping)
4. Selects best available proxy (with retry on failure)
5. Connects to target through proxy (CONNECT for HTTP/HTTPS or SOCKS5)
6. Bidirectional relay (`tokio::io::copy_bidirectional`)
7. Optionally pools connection for reuse
### **Sticky Sessions:**
- Client IP → proxy key affinity
- TTL: configurable (default 300s)
- HTTPS-only checks TLS-clean flag on reuse
- Auto-cleared on proxy failure (retry with fresh selection)

## **10. KEY DESIGN PATTERNS**
### **Fast-Path Optimization:**
- Level 1 fast bootstrap allows proxy server to start before full pool is ready
- `first_ready` notify enables waiting clients to proceed immediately
- No blocking on pool population
### **Graceful Degradation:**
- Tier-based proxy selection (verified → presumed → unchecked)
- HTTPS falls back to non-TLS-clean proxies if needed
- Retry logic (up to 3 attempts per request)
### **Persistence:**
- State.json saves proxy metadata between restarts
- Level 0 restores trusted proxies instantly
- Reduces convergence time on restart
### **Scalability:**
- Lock-free DashMap for 10k+ proxies
- Atomic counters instead of mutexes
- Bounded GeoIP concurrency (20 requests max)
- Connection pooling reduces TCP overhead

       gpush "fix: описание"

  4. Смержить PR на GitHub:

  После merge в main:
  - CI запускается автоматически
